CRITICAL — must be resolved before accepting real user submissions
================================================================================

[ ] Provision the Supabase Postgres dev database (REQUIRED to run submit)
    - Drizzle schema, migrations, and DataStore implementation are all
      in place. The app just needs a DATABASE_URL to point at.
    - Steps:
        1. Create a Supabase project at https://supabase.com/dashboard.
        2. Project Settings → Database → Connection string → "URI".
        3. Paste the URI into .env.local as DATABASE_URL=…
        4. Run `npm run db:migrate` to create the tables.
        5. Submit a test bracket end-to-end.
    - Per AGENTS.md, Supabase is for DB only (no Auth/Storage/Functions).
    - The same schema and migrations will apply to AWS RDS / Aurora
      Postgres in production — swap DATABASE_URL, re-run migrations.


[ ] Proxy the Alchemy API key through a server-side API route
    - Current: NEXT_PUBLIC_ALCHEMY_API_KEY is exposed to the browser bundle,
      so anyone can read it out of the JS and use it against our quota.
    - Fix: Move the getNFTsForOwner call in src/lib/web3/alchemy.ts behind
      a new /api/alchemy/tokens route. Keep the key server-only.

[ ] Proper auth layer on /api/submit and /api/leaderboard
    - Today any client can POST to /api/submit with an arbitrary wallet
      or email and no signed proof of ownership. A malicious user on
      another site could construct a form that submits a bracket for a
      victim's wallet address.
    - /api/leaderboard returns raw identifiers (wallets are fine, but
      email plaintext leaks to the network tab / localStorage / anyone
      scraping the endpoint). Frontend client-side-truncates for display
      but the raw value is still served.
    - Proper fix: signed wallet message (SIWE / eth_signTypedData) on
      submit for wallets, email confirmation link for email identities.
      On /api/leaderboard, replace `identifier` with an opaque owner
      key (short stable hash) so the "You" row can still match without
      leaking the email. See comment in src/app/api/leaderboard/route.ts.

[ ] Smoke-test the delegate.xyz v2 integration end-to-end
    - Use a real pair: cold wallet holds the Meebit Futbol NFTs, hot wallet
      is delegated via https://delegate.xyz/ (ALL / CONTRACT / ERC721
      delegation types all need to work).
    - Sign in with the hot wallet and verify:
        a) the delegated NFT count appears in useUserDivision
        b) the welcome modal shows "X owned + Y delegated"
        c) the user lands in the correct division
    - Code is live (src/lib/web3/delegateRegistry.ts + useDelegatedBalance.ts)
      but has never been exercised against the real on-chain registry.

[ ] Confirm gating-contract states on mainnet
    - Meebits Futbol (0x8fBb…f4E9): Bronze / Silver gate. If not yet
      minted, Bronze and Silver are empty until mint lands.
    - Meebits (0x7Bd2…6Bc7): Gold gate. The original Meebits collection
      is fully minted and actively traded, so Gold works from day one.
    - Decide whether launching with only Gold + Free active pre-mint is
      acceptable, or hold until Meebits Futbol is live.

[ ] Sanity-check the tournament lock date
    - Hardcoded in src/lib/divisions.ts:
        TOURNAMENT_LOCK_DATE = new Date('2026-06-11T16:00:00Z');
    - After this moment, isTournamentLocked() blocks new submissions
